# <low-emphasis>atom.io</low-emphasis>/eslint-plugin

`atom.io/eslint-plugin` provides rules that keep atom.io state and transaction
declarations explicit, consistent, and safe around typed errors.

## install

If your project already depends on `atom.io`, the plugin is already available from that
package. For a minimal TypeScript flat config, you also need ESLint and
`typescript-eslint`:

<CodeBlock
filepath="sample.sh"
label="install eslint"
code={`pnpm add -D eslint typescript-eslint`}
/>

## minimal flat config

Here is a small `eslint.config.ts` that starts from `typescript-eslint`'s recommended
type-checked rules and adds the atom.io rules.

<Exhibit src="tooling/eslint-plugin/eslint.config.ts.txt" />

Why use `recommendedTypeChecked` instead of plain `recommended`?

`atom.io/exact-catch-types` needs TypeScript type information so it can compare an
atom's error type with its `catch` constructors. If you want a syntax-only ESLint setup,
use `typescript-eslint`'s plain recommended config and leave out
`atom.io/exact-catch-types`.

## rules

### explicit-state-types

`atom.io/explicit-state-types` requires state declarations to include explicit type
arguments.

<Exhibit src="tooling/eslint-plugin/explicit-state-types.ts" />

This keeps important state types visible at the declaration site, where they are easiest
to audit.

### explicit-transaction-types

`atom.io/explicit-transaction-types` requires transaction declarations to include
explicit function type arguments.

<Exhibit src="tooling/eslint-plugin/explicit-transaction-types.ts" />

This keeps transaction inputs and outputs visible without having to inspect the
implementation.

### naming-convention

`atom.io/naming-convention` keeps declaration variable names and state keys aligned.

<Exhibit src="tooling/eslint-plugin/naming-convention.ts" />

For example, `countAtom` should use `key: "count"`, `historyTimeline` should use
`key: "history"`, and `saveTransaction` should use `key: "save"`. Families use
plural suffixes, so `userAtoms` uses `key: "user"` and `historyTimelines` uses
`key: "history"`. The same pattern applies to selectors and mutable atoms.

### exact-catch-types

`atom.io/exact-catch-types` checks that a state declaration's typed error channel
matches the constructors listed in its `catch` option.

<Exhibit src="tooling/eslint-plugin/exact-catch-types.ts" />

This helps prevent state from advertising that it may catch one error type while actually
catching another.
